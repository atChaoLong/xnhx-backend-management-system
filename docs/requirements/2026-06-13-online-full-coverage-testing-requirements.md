# 线上全覆盖测试方案

## Problem Frame

当前线上地址为 `https://xiaoniuhaoxue.paitongai.cn`。已有线上测试主要覆盖了公开入口、未登录保护、登录失败、健康检查和教师公开表单的匿名访问，结论是基础入口和匿名保护可用，但真实后台业务流、角色权限矩阵、ClassIn 写入链路、核心 CRUD 和近期补齐功能仍缺少系统化线上验收。

本方案用于把线上测试从“点状冒烟”升级为“可重复、可追踪、可阻断发布”的全覆盖验收。测试目标不是一次性证明系统完美，而是明确哪些能力必须在线上可用，哪些风险必须被拦住，哪些场景可以延期但必须有记录。

## Stakeholders

- 管理员：验证账号、权限、菜单、字典、全局配置和数据可见性。
- 销售团队：验证线索、试听、转正、跟进和订单相关流程。
- 教务团队：验证学生档案、课时、排课、班级、课程和 ClassIn 课堂流程。
- 招师团队：验证候选老师、面试、评分、复核、入库和教师公开表单流程。
- 财务团队：验证订单、流水、退款、余额和敏感字段可见性。
- 老师/外部填表用户：验证公开教师申请表、附件上传和提交结果。
- 开发/运维：验证部署、环境变量、数据库迁移、日志、安全边界和故障回滚。

## Requirements

### R1. 环境与部署准入

- 测试开始前必须记录线上域名、Vercel 部署 ID、Git commit、构建时间和 Supabase 项目环境。
- `/api/health` 必须返回 200，并记录响应内容。
- 必须确认数据库迁移已应用到线上，尤其是近期新增的老师、学生、试听、财务、质量报告、ClassIn callback 相关表和字段。
- 必须确认线上环境变量、Supabase Storage bucket、ClassIn 凭证和公开表单配置已就绪。
- 如果线上仍未部署最新代码，测试只能进入“当前线上版本验收”，不能标记新功能通过。

### R2. 测试账号矩阵

- 至少准备管理员、销售、教务、财务、招师/HR、普通老师或受限账号、禁用账号各一套。
- 每个账号必须记录角色、预期菜单、预期 API 权限、预期数据范围。
- 必须包含“有 auth 用户但缺失 profile”“profile 禁用”“角色权限不足”三类负向账号或等价模拟。
- 管理员账号只用于基线和配置验证，不能替代普通角色验收。

### R3. 公开入口与匿名访问

- 验证首页、登录页、教师公开表单、教师表单成功页的访问和视觉可用性。
- `/signup` 在生产环境默认关闭时应返回 404；如果启用 `ENABLE_PUBLIC_SIGNUP=true`，则必须按注册流程验收。
- `/debug/auth` 在生产环境默认应不可公开访问；如临时启用，必须只允许授权管理员访问。
- 匿名访问后台页面必须跳转登录或返回受控错误，不能泄露业务数据。
- 匿名访问 API 必须只允许白名单公开接口，例如健康检查和教师公开表单所需的专用字典接口。

### R4. 登录、会话与用户档案

- 验证有效登录、无效登录、退出登录、刷新页面、会话过期和 token refresh。
- `/api/auth/signin` 不应再出现 `PROFILE_LOOKUP_FAILED` 的正常账号失败；如果出现，必须记录账号、profile 状态和后端错误。
- 缺失 profile、禁用账号、角色无效账号必须 fail closed，并给出用户可理解的错误。
- 登录后 `/api/auth/session` 和 `/api/auth/profile` 返回的数据必须最小化，不泄露不该给前端的敏感字段。

### R5. 角色权限与数据范围

- 每个角色必须验证菜单可见性、页面直达、API 直调、数据列表、详情页、创建、编辑、删除或状态流转。
- 权限不足时，页面和 API 应返回一致的 401/403/404 语义，不能只靠前端隐藏按钮。
- 必须覆盖 IDOR 场景：用低权限账号直接访问其他销售、其他教务或其他老师关联的数据 ID。
- 敏感字段必须按角色脱敏，例如手机号、财务金额、ClassIn 账号密码、老师薪资/评级等。

### R6. 核心业务闭环

- 线索：创建、编辑、领取、释放、跟进、状态变化、来源字段、重复数据处理。
- 试听：从线索生成试听、试听列表、试听详情、老师/学生关联、状态流转、开课前检查。
- 转正：试听转正式订单、正式学生生成、课程生成、余额和课时计算。
- 学生：学生列表、详情、编辑、回访、状态历史、课时消耗、班主任分配。
- 排课：批量排课、预检查、课程/课节生成、日历展示、冲突处理。
- 反馈与待办：课堂反馈、待办创建、完成、SLA 或逾期状态。
- 质量与报表：服务质量、试听转化、质量报告创建和权限控制。

### R7. 招师与教师流程

- 候选老师：创建、编辑、列表、详情、状态流转和附件关联。
- 面试流程：约面安排、面试记录、视频上传、评分、复核、入库、薪资/级别/素质评价。
- 教师库：老师列表、详情、教学视图、销售视图、异常记录和 ClassIn 注册。
- 公开表单：令牌校验、字典加载、附件上传、提交、重复提交保护、成功页。
- 教师敏感字段必须按角色脱敏，公开接口不得复用后台受保护字典接口。

### R8. ClassIn 集成

- 验证学生、老师、课程、教室、课节的同步、注册、关联和重建入口。
- 验证 `open-class`、`create-classin`、批量创建 ClassIn 课堂、从订单创建课程等关键写入流程。
- 验证 ClassIn callback 的签名、幂等、事件落库、失败重试和异常日志。
- 涉及真实第三方写入时必须使用专用测试账号、测试课程和测试班级，并记录外部 ID。
- 第三方失败时，系统必须给出可恢复状态，不能造成内部数据半成功且无记录。

### R9. 财务与敏感数据

- 验证正式订单、交易流水、退款/撤销、支付凭证、余额、课时金额和关联实体。
- 非财务角色不得看到完整敏感财务字段。
- 创建或修改财务数据必须使用专用测试订单，不能使用真实客户订单。
- 金额、课时、退款、余额变动必须有可追踪事件或审计字段。

### R10. 上传、文件与安全边界

- 验证教师表单附件、支付凭证、通用上传接口的大小、类型、鉴权和存储路径。
- 未登录用户只能访问明确允许的公开上传入口。
- 禁止可执行文件、超大文件、伪装 MIME 文件和跨用户读取附件。
- 测试截图和导出报告必须脱敏，不保存真实手机号、密码、token 和客户隐私。

### R11. 前端兼容与可用性

- 至少覆盖桌面宽屏、普通笔记本宽度和移动宽度。
- 核心页面不得出现按钮文字溢出、表格遮挡、弹窗不可关闭、表单无法提交、加载状态卡死。
- 浏览器控制台不得有阻断流程的运行时错误。
- 页面刷新、后退、重复提交和网络慢速场景下不得破坏数据一致性。

### R12. 性能与可靠性基线

- 关键列表页和详情页应记录首屏可交互时间和主要 API 响应时间。
- 登录、列表、详情、创建、状态流转等核心接口应有基础响应时间阈值；阈值可先按实测基线设定。
- 重复刷新、连续操作、并发提交相同动作时，应验证幂等或明确错误提示。
- 外部依赖异常、网络超时、Vercel 冷启动或 Supabase 错误必须有可理解的前端反馈。

### R13. 缺陷记录与复测闭环

- 每个缺陷必须记录环境、账号角色、测试数据 ID、复现步骤、期望结果、实际结果、截图或接口响应、严重级别、负责人和状态。
- P0/P1 修复后必须复测原路径、相邻路径和权限负向路径。
- 每轮测试必须产出一份报告，建议路径为 `.gstack/qa-reports/online-full-coverage-YYYY-MM-DD.md`。
- 如果发现线上部署与本地代码不一致，报告必须单独列为部署阻塞，而不是归因到功能失败。

## Success Criteria

- P0/P1 场景全部通过，或有明确业务负责人签字接受的延期记录。
- 未登录、低权限、跨数据范围访问不能读取、修改或导出受保护数据。
- 管理员、销售、教务、财务、招师至少各完成一条核心可用路径。
- 线索到试听、试听到转正、学生排课、教师公开表单到候选老师、候选老师到入库至少各跑通一条线上测试数据。
- ClassIn 相关写入流程在测试账号范围内可成功或可解释失败，并且内部状态可恢复。
- 线上测试报告包含通过项、失败项、阻塞项、未测项和复测结论。

## Scope Boundaries

- 本方案不包含大规模压测，只做可靠性和响应时间基线。
- 本方案不使用真实客户订单做破坏性测试。
- 本方案不验证第三方 ClassIn 平台自身 SLA，只验证本系统对第三方成功、失败和回调的处理。
- 本方案不要求本轮立刻完成自动化测试实现，但所有手工测试证据应按自动化可复用的结构记录。
- 本方案不把生产默认关闭的 `/signup` 和 `/debug/auth` 视为缺陷，除非业务明确要求线上开放。

## Key Decisions

- 采用“线上手工全覆盖 + 可自动化证据沉淀”的方式先补齐验收，不等待自动化框架完成。
- 测试按层推进：部署准入、匿名安全、登录会话、角色权限、业务闭环、第三方集成、回归复测。
- 所有新增或变更数据必须带测试标识，避免污染真实业务数据。
- 管理员只能用于配置和基线验证，角色权限必须用对应角色账号实测。
- ClassIn 写入类测试默认需要业务确认测试账号和测试课程，未经确认不对真实第三方资源做破坏性操作。

## Dependencies

- 可用的线上管理员账号和各角色测试账号。
- 可安全创建、修改、清理的测试线索、学生、订单、老师、课程和 ClassIn 资源。
- 最新代码已部署到 `xiaoniuhaoxue-next` 对应 Vercel 项目，并能通过域名访问。
- Supabase 线上数据库迁移、Storage bucket、RLS/服务端权限和环境变量已配置。
- ClassIn 测试凭证或明确允许使用的线上测试资源。
- 截图、接口日志、浏览器控制台和网络请求证据采集方式。

## Risk Assessment

- 误改真实数据：使用专用测试记录、测试前记录 ID、涉及删除/退款/外部写入前二次确认。
- 权限漏测：按角色矩阵逐项跑页面直达和 API 直调，不用管理员结果代替。
- 部署不一致：每轮测试先记录部署 ID、commit 和关键修复是否上线。
- 第三方写入不可逆：ClassIn 操作使用测试账号和测试班级，必要时只验证到预检查。
- 敏感信息泄露：报告、截图、日志统一脱敏，不记录 token、密码、完整手机号。
- 线上环境不稳定：把网络、Vercel、Supabase、ClassIn 异常单独标记为阻塞或外部依赖失败。

## Rollout Plan

### Phase A. 测试准备

- 确认部署版本、域名、健康检查、数据库迁移和关键环境变量。
- 准备账号矩阵、测试数据命名规则、测试记录表和回滚/清理策略。
- 明确 ClassIn 写入边界和公开注册/调试页的线上策略。

### Phase B. 匿名与安全冒烟

- 扫描公开页面和匿名 API。
- 验证后台页面未登录跳转、受保护 API 401/403、公开教师表单可用。
- 复测已知问题：公开表单字典接口、`/signup` 生产策略、`/debug/auth` 生产策略、Supabase 多客户端警告。

### Phase C. 登录与权限矩阵

- 使用每个角色登录，验证菜单、页面、API、数据范围和敏感字段。
- 覆盖直接访问 URL、复制其他角色数据 ID、禁用账号、缺失 profile 等负向路径。

### Phase D. 业务主链路

- 跑通线索、试听、转正、正式学生、课程、课节、反馈、待办、质量报告和财务流水。
- 每条链路保留测试数据 ID、截图和接口证据。

### Phase E. 招师与 ClassIn 集成

- 跑通教师公开表单、候选老师、面试、评分、复核、入库和教师库流程。
- 跑通或受控验证 ClassIn 注册、同步、课堂创建、回调和失败恢复。

### Phase F. 缺陷修复与回归

- 按 P0/P1/P2 分级修复。
- 每个修复至少复测原缺陷、相邻正向路径和对应负向权限路径。
- 输出最终线上全覆盖报告和剩余风险清单。

## Open Questions

- 需要提供哪些线上测试账号？至少要包含管理员、销售、教务、财务、招师、受限角色和禁用账号。
- ClassIn 线上测试是否允许真实写入？如果允许，使用哪个测试学校、课程、老师和学生账号？
- `/signup` 是否保持生产关闭？如果要开放，需要补注册全链路测试。
- `/debug/auth` 是否保持生产关闭？如果要开放，需要确认只允许管理员访问。
- 测试数据是否允许由测试人员创建并清理？哪些表或真实客户数据绝对不能触碰？
- Vercel 的当前线上部署 ID 和 Git commit 以哪个来源为准？

## Recommended Next Step

先执行 Phase A 和 Phase B，确认线上部署、健康检查、匿名安全和公开教师表单已经可用。若 Phase A/B 通过，再进入账号矩阵和业务主链路；若不通过，优先修复部署或公开接口问题，避免在旧版本上浪费全覆盖测试时间。
