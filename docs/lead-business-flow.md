## 📋 完整业务流程

### 流程图

```
┌─────────────────────────────────────────────────────────────┐
│                      线索生命周期                              │
└─────────────────────────────────────────────────────────────┘

1. 运营录入线索
   ↓
   状态: 运营未派单 (unassigned)
   条件: grab_wechat 为空
   操作:
   - 运营: 填写 grab_wechat 派单给销售

2. 销售接单 (运营派单后)
   ↓
   状态: 销售未反馈 (waiting_feedback)
   条件: grab_wechat 有值，add_status 为空
   操作:
   - 销售: 反馈"已添加"或"未添加"

3. 销售反馈后
   ↓
   分支A: 反馈"已添加"
   状态: 已添加 (added)
   条件: add_status = 'added'
   操作:
   - 销售或班主任: 创建试听课程 ✅

   分支B: 反馈"未添加"
   状态: 未添加 (not_added)
   条件: add_status = 'not_added'
   操作:
   - 无后续操作，线索结束

4. 创建试听后
   ↓
   状态: 试听 (trial)
   条件: trial_lessons 表中有记录
   操作:
   - 教务: 匹配老师
   - 教务: 确认老师
   - 教务: 确定时间
   - 教务: 生成链接

5. 试听完成
   ↓
   分支A: 转化成功
   状态: 正式 (formal)
   条件: formal_orders 表中有记录
   操作:
   - 销售: 录入正式订单

   分支B: 未转化
   状态: 空 (empty)
   条件: 没有订单记录
   操作:
   - 线索结束
```

## 🎯 关键业务规则

### 规则1: 创建试听的前置条件 ✅

**业务规则**: 只有线索状态为"已添加"时，才能创建试听课程

**原因**:
- 销售反馈"已添加" = 已和家长建立联系
- 此时才有必要创建试听课程
- "未添加"的线索不需要试听

**实现代码**:
```typescript
// app/dashboard/leads/page.tsx:320
{leadsPerm.convert() && lead.add_status === 'added' && (
  <Button onClick={() => handleCreateTrialLesson(lead)}>
    创建试听
  </Button>
)}
```

**状态对照**:

| 线索状态 | add_status | 能否创建试听 | 说明 |
|---------|-----------|------------|------|
| 运营未派单 | null 或 grab_wechat为空 | ❌ | 还未派给销售 |
| 销售未反馈 | null | ❌ | 销售还未反馈 |
| 已添加 | 'added' | ✅ | **可以创建试听** |
| 未添加 | 'not_added' | ❌ | 家长未添加，不需要试听 |

---

### 规则2: 反馈按钮显示条件 ✅

**业务规则**: 反馈按钮只对派给当前销售自己的线索显示

**原因**:
- 销售只能操作派给自己的线索
- 避免误操作其他销售的线索
- 职责清晰

**实现代码**:
```typescript
// app/dashboard/leads/page.tsx:307
{leadsPerm.feedback() && lead.grab_user_id === user?.id && (
  <Button onClick={() => handleMarkAsFeedback(lead)}>
    反馈
  </Button>
)}
```

**场景示例**:

| 用户 | 线索所属 | 反馈按钮显示 | 说明 |
|-----|---------|------------|------|
| 销售A | 销售A | ✅ 显示 | 可以操作自己的线索 |
| 销售A | 销售B | ❌ 不显示 | 不能操作他人的线索 |
| 班主任 | 任何人 | ❌ 不显示 | 班主任不负责反馈 |
| 运营 | 任何人 | ❌ 不显示 | 运营不负责反馈 |

### 规则3: 试听状态流转 ✅

**试听课程8种状态**:

```typescript
// lib/status-calculator.ts:133-142
export enum TrialLessonStatus {
  CANCELLED = 'cancelled',           // 1. 取消试听
  WAITING_MATCH = 'waiting_match',   // 2. 待匹配老师
  WAITING_CONFIRM = 'waiting_confirm', // 3. 待确认老师
  WAITING_TIME = 'waiting_time',     // 4. 待确认时间
  WAITING_LINK = 'waiting_link',     // 5. 待开链接
  SCHEDULED = 'scheduled',           // 6. 已排待上课
  WAITING_FEEDBACK = 'waiting_feedback', // 7. 上完待反馈
  COMPLETED = 'completed'            // 8. 已完成
}
```

**状态流转条件**:

| 当前状态 | 流转条件 | 下一状态 | 触发角色 |
|---------|---------|---------|---------|
| 待匹配老师 | 匹配老师字段有值 | 待确认老师 | 教务 |
| 待确认老师 | 确认老师字段有值 | 待确认时间 | 教务 |
| 待确认时间 | 确定时间字段有值 | 待开链接 | 教务 |
| 待开链接 | 上课链接字段有值 | 已排待上课 | 教务 |
| 已排待上课 | 上课时间已过 | 上完待反馈 | 系统 |
| 上完待反馈 | 填写是否转化 | 已完成 | 老师/销售 |

## 🔐 权限矩阵详细说明

### 线索管理 (leads)

| 操作 | 运营 | 销售 | 班主任 | Admin | 说明 |
|-----|-----|------|--------|-------|------|
| **view** | ✅ | ✅ | ✅ | ✅ | 所有角色都可以查看 |
| **create** | ✅ | ❌ | ❌ | ✅ | 只有运营和管理员可以录入 |
| **edit** | ✅ | ❌ | ❌ | ✅ | 只有运营和管理员可以编辑 |
| **delete** | ✅ | ❌ | ❌ | ✅ | 只有运营和管理员可以删除 |
| **feedback** | ❌ | ✅* | ❌ | ❌ | 只有销售可以反馈，且只能反馈派给自己的 |
| **convert** | ❌ | ✅† | ✅† | ❌ | 销售/班主任可以创建试听，但仅在"已添加"状态 |

\* 销售只能反馈派给自己的线索（grab_user_id === user.id）
† 只在线索状态为"已添加"时显示创建试听按钮（add_status === 'added'）

### 线索列表页按钮显示条件

```typescript
// 1. 新增线索按钮
{leadsPerm.create() && (
  <Link href="/dashboard/leads/new">
    <Button>新增线索</Button>
  </Link>
)}
// 显示角色: 运营、Admin
// 显示条件: 无额外条件

// 2. 反馈按钮
{leadsPerm.feedback() && lead.grab_user_id === user?.id && (
  <Button onClick={handleMarkAsFeedback}>反馈</Button>
)}
// 显示角色: 销售
// 显示条件:
//   1. 有feedback权限
//   2. 线索派给当前用户 (grab_user_id === user.id)
// 禁用条件: add_status === 'added' (已反馈后禁用)

// 3. 创建试听按钮
{leadsPerm.convert() && lead.add_status === 'added' && (
  <Button onClick={handleCreateTrialLesson}>创建试听</Button>
)}
// 显示角色: 销售、班主任
// 显示条件:
//   1. 有convert权限
//   2. 线索状态为"已添加" (add_status === 'added')

// 4. 编辑按钮
{leadsPerm.edit() && (
  <Link href={`/dashboard/leads/${lead.id}/edit`}>
    <Button>编辑</Button>
  </Link>
)}
// 显示角色: 运营、Admin
// 显示条件: 无额外条件

// 5. 删除按钮
{leadsPerm.delete() && (
  <Button onClick={handleDeleteClick}>删除</Button>
)}
// 显示角色: 运营、Admin
// 显示条件: 无额外条件
```

## 🎯 核心设计原则

### 1. 职责分离
- **运营**: 录入和派单
- **销售**: 反馈和创建试听
- **班主任**: 创建试听
- **教务**: 匹配老师和排课
- **Admin**: 系统管理，不参与业务

### 2. 状态驱动
- 所有操作按钮都根据状态显示
- 状态决定下一步可以做什么
- 避免非法状态流转

### 3. 权限最小化
- 每个角色只能看到自己的操作按钮
- 不能操作他人的数据
- Admin也受权限限制

### 4. 数据一致性
- grab_user_id 必须等于 user.id 才能反馈
- add_status 必须等于 'added' 才能创建试听
- 通过试听表关联到线索，确保转化状态准确
